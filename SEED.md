# SEED.md

**This repository is a seed.** It is the **canonical-grounded** Rust-port
target for `AdaWorldAPI/redmine` (the upstream Ruby/Rails source), and the
Redmine-side sibling of `AdaWorldAPI/openproject-nexgen-rs`.

## Canonical-first, not structure-first

`openproject-nexgen-rs` was seeded *structure-first*: it mirrored an existing
manual Rust port (`openproject-rs`) to harvest correct Rust idioms (crate
layout, axum handlers, SQLx models) as a codegen TargetSpec.

`redmine-rs` is seeded **canonical-first**. There is no manual Redmine port to
mirror — instead the foundation is the **OGAR canonical extraction** of the
Redmine corpus. The first thing that lands is the *codebook contract*
(`redmine-canon`); the Rust idioms come later, and can be borrowed wholesale
from `openproject-nexgen-rs` because **both ports speak the same codebook**.
A Redmine `Issue` and an OpenProject `WorkPackage` are the same canonical node
(`project_work_item`, `0x0102`), so the generated Rust types are shared shapes,
not parallel reimplementations.

This is the convergence thesis made operational: the fork lineage
(Redmine → ChiliProject → OpenProject) preserved the invariant, OGAR proved it
on the real corpora, and the two `-rs` ports inherit it.

## The pipeline

```text
  AdaWorldAPI/redmine (Ruby/Rails)
        │  ruff_ruby_spo::extract_with(path, "redmine")
        ▼
  ruff_spo_triplet::ModelGraph
        │  ogar_from_ruff::lift_model_graph        (domain-gated: a project
        │                                            curator yields only 0x01 ids)
        ▼
  Vec<ogar_vocab::Class>   →  canonical_concept + canonical_id (CODEBOOK)
        │  snapshot dump
        ▼
  crates/redmine-canon/data/redmine.ogar.json
```

## What changes vs the seed

1. **More crates.** Domain crates (`rm-work-items`, `rm-actors`,
   `rm-projects`, …) layered on the `redmine-canon` codebook ids — ideally
   sharing canonical type definitions with `openproject-nexgen-rs`.
2. **A richer snapshot.** As OGAR promotes more concepts, the snapshot is
   regenerated (it grows; ids are stable forever — never re-assigned). The
   per-concept canonical *shape* (typed attributes, family edges) can be added
   alongside the mapping table.
3. **A regeneration path.** A small `xtask`/CI step that re-runs the OGAR
   producer against a pinned Redmine checkout and rewrites the snapshot, so
   drift between OGAR's codebook and this repo is caught mechanically.

## What NOT to do

- **Do not hand-edit `data/redmine.ogar.json`.** It is generated by the OGAR
  producer. If a mapping is wrong, the fix lives in `ogar-vocab` (the codebook
  / resolver) or the extractor — not in the vendored snapshot.
- **Do not re-mint ids.** Codebook ids are minted in `AdaWorldAPI/OGAR` and are
  stable forever. This repo *consumes* them; it never invents them.
- **Do not diverge the canonical types from `openproject-nexgen-rs`.** Where
  both ports realise the same concept, the Rust shape should be shared. The
  point of the codebook is that they cannot drift.
- **Do not commit code that doesn't compile.** Every change ends with
  `cargo test` green.

🦋 The seed is the starting line, not the finish.
